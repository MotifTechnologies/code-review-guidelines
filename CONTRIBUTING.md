# Contributing to Code Review Process Guidelines

Thank you for your interest in contributing to this project! This document provides guidelines for contributing.

## How to Contribute

### Reporting Issues

If you find a bug, have a question, or want to suggest an improvement:

1. Check existing issues to avoid duplicates
2. Create a new issue using the appropriate template:
   - **Bug Report**: For errors or problems with the guidelines
   - **Feature Request**: For suggesting new content or improvements
   - **Process Improvement**: For suggesting changes to how we work

### Submitting Changes

1. **Fork the repository**
2. **Create a branch** following our naming convention:
   - `docs/description` - Documentation changes
   - `feature/description` - New features or content
   - `fix/description` - Bug fixes
   - `experiment/username/description` - Experimental work

3. **Make your changes**
   - Follow existing formatting and style
   - Update relevant documentation
   - Add yourself to contributors if this is your first contribution

4. **Submit a Pull Request**
   - Use the appropriate PR template (SHIP, SHOW, or ASK)
   - Provide a clear description of your changes
   - Reference any related issues

## PR Tier Guidelines

When submitting a PR, choose the appropriate tier:

### SHIP Tier (Auto-merge eligible)
- Typo fixes
- Minor documentation updates
- Formatting changes

### SHOW Tier (Async review)
- New examples or templates
- Content additions that don't change core guidelines
- Tool configuration updates

### ASK Tier (Full review required)
- Changes to core review guidelines
- New review tiers or categories
- Significant restructuring

## Content Guidelines

### Writing Style
- Use clear, concise language
- Prefer active voice
- Include practical examples
- Link to external resources when helpful

### Formatting
- Use Markdown formatting consistently
- Include code blocks with language hints
- Use tables for comparison data
- Add diagrams (Mermaid) for complex workflows

### Examples
- Provide real-world, practical examples
- Include both good and bad examples when relevant
- Test any code/config examples before submitting

## Review Process

All contributions go through our tiered review process:

1. **Automated checks** run on all PRs
2. **Tier classification** determines review requirements
3. **Human review** for SHOW and ASK tier PRs
4. **Merge** after approval and CI passes

## Code of Conduct

### Our Standards

- Be respectful and inclusive
- Welcome newcomers
- Focus on constructive feedback
- Accept feedback gracefully

### Unacceptable Behavior

- Harassment or discrimination
- Trolling or inflammatory comments
- Personal attacks
- Publishing others' private information

## Getting Help

- **Questions**: Open an issue with the "question" label
- **Discussions**: Use GitHub Discussions for broader topics
- **Direct contact**: Reach out to maintainers for sensitive issues

## Recognition

Contributors are recognized in:
- The README contributors section
- Release notes for significant contributions
- Our internal documentation

Thank you for helping improve code review practices for everyone!
